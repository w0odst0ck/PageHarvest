# PageHarvest — 产品路线规划

> 从 CLI 工具包到全流程应用产品。

---

## 现状（v0.x — CLI 工具包）

- 命令行驱动，对 Python 和命令行有一定要求
- 工作流分 3~5 步，手动串联
- 详情页采集依赖人工浏览器操作
- 数据结果以 CSV / Excel / JSON 输出

**目标用户：** 我自己、能跑 Python 的开发者

---

## 愿景

让非技术用户也能完成"搜索→选品→详情→比对"的全流程。
核心价值不是采集技术，而是**选品决策链路的数据化**。

---

## 路线图

### Phase 1: 体验提升（当前 → 短期）

**目标：** 降低使用门槛，减少手工程度

| 方向 | 方案 | 说明 |
|------|------|------|
| Web UI | Streamlit 面板 | 快速出 MVP，数据可视化 |
| 一键工作流 | 单个命令跑完所有阶段 | 串联选品→详情→缺口 |
| 本地分析报告 | HTML 报告替代终端输出 | 可视化 + 可分享 |
| 自动路径发现 | 智能识别 data/ 目录结构 | 减少配置参数 |

**具体：**
```bash
# 目标是简化到这种程度
pageharvest run --input data/ZKH/
# 自动检测平台、自动发现选品数据、自动跑完所有阶段
```

### Phase 2: 采集辅助（中期）

**目标：** 减少人工浏览器操作

| 方向 | 方案 | 说明 |
|------|------|------|
| 浏览器扩展 | 一键保存当前页商品数据 | 替代 Ctrl+S，数据直接入库 |
| 批量下载工具 | 完善 Playwright 方案 | 处理 WAF，自动排队 |
| 油猴脚本增强 | 搜索页自动翻页+保存 | 批量采集搜索页 |

### Phase 3: 桌面应用（中长期）

**目标：** 独立产品，非技术用户可用

| 方向 | 方案 | 说明 |
|------|------|------|
| 技术栈 | Tauri（Rust + Web） | 跨平台、小体积 |
| 核心功能 | 项目管理、平台配置、工作流编排 | |
| 渲染方案 | 内嵌 Chromium 处理 WAF | 采集辅助 |
| 数据视图 | 表格 / 图表 / 报告预览 | |
| 扩展 | 插件市场：平台适配器、选品策略、报告模板 | |

**可选替代方案：**
- Electron（体积大但生态成熟）
- 纯 Web（SaaS 模式，但涉及采集合规问题）

UI 层作为独立的前端，通过 API 或子进程调用核心。

## 关键设计决策

### 数据存储
- **本地优先**，不上传用户数据
- 项目化：一个项目 = 一个目录，含配置 + 数据 + 结果
- 可选云同步（未来）

### 平台适配器
- 保持 Plugin 模式不变
- 社区可贡献新平台
- 通过 `pip install pageharvest-{platform}` 分发

### 详情页采集
- **默认离线**：人工保存 HTML → 本地解析（零风险）
- **可选自动化**：Playwright 渲染（标记为实验性）
- 浏览器扩展辅助（中期）

### 并发与合规
- 不提供高速抓取能力
- 尊重 robots.txt 和平台条款
- 采集辅助仅辅助人工操作，非自动化爬虫

## 不变的原则

1. **离线优先** — 不上线采集器，不碰反爬红线
2. **核心 CLI 一直可用** — UI 是附加层，不依赖 UI 也能用
3. **数据属于用户** — 不上传、不收集、不做 SaaS
4. **插件可扩展** — 平台、策略、报告模板都可插拔
